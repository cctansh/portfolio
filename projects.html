Full-Stack Banking Website

docker • expressjs • postgresql • nextjs • tailwindcss

a mock banking website made for OCBC's challenge statement, "Digital Banking Accessible for Everyone".

working in a team of 5, we created a docker project utilising a locally hosted postgresql database, expressjs for the backend API, and nextjs + tailwindcss for the frontend.

i setup the database, in addition to creating the backend api attached to it. i also assisted in connecting the frontend to the backend api, using GET calls to fetch and display all relevant data.

below are several notable accessibility features i implemented:

speech-to-text

utilised webkitSpeechRecognition to enable voice input functionality. event handlers such as onresult, onstart, onend were used to capture the speech result, and update the UI to indicate whether speech is currently being recorded. key listeners used to allow users to toggle speech recognition using the keyboard, for further accessibility.

voice navigation

similar to the speech-to-text function, webkitSpeechRecognition, event handlers, and key listeners were used to handle recognising the speech. afterward, router is used to navigate to the relevant page, based on the speech recognition.

adjustable text sizes

react context api is used to manage the state of the global text size variable across all pages. the method GetTextSizeClass is exported and can be used to wrap all relevant text in the code. when toggling the text size options in the header, the method replaces the tailwindcss font size classes with the appropriate font size class.

qr code authenticator

in the backend, speakeasy is used to generate a unique secret key for the user, which is then stored in the database and is used to generate TOTP tokens. these tokens can be validated by the speakeasy library. qrcode (nodejs) is used to convert the otpauth:// URL generated by speakeasy into a QR code that can be displayed on the frontend, and is scannable by an authenticator to generate the TOTP token.

the frontend profile page calls the method to generate the qr code, which is hidden behind the user's password verification for additional security. any sensitive data (transaction history) or actions (making a transaction) will then call the MFAModal component, which contains a text field to enter the generated TOTP token, and checks it against the backend to ensure it is a valid token before allowing the user to proceed.

Event Ticketing Mobile Application

android studio • java • firebase • geocoder • google places api

a mock event ticketing mobile application created for a mobile app development module.

working in a team of 5, we used android studio to build the application using java, connected to a firebase database.

i assisted in retrieving the event details from the database, in addition to creating the event search and filter page.

below are additional features i created, with relation to the event venue page:

venue details

the venue address is retrieved from the event object using its getter function. afterward, the geocoding api is called on it to retrieve the locations placeID. this placeID is then passed into the google places api call, which retrieves all relevant venue details. textViews and cardViews in the frontend are then updated with the retrieved details.

google maps fragment

utilises the inbuilt supportMapFragment to load in a google maps visual. the geocoding api is once again used to retrieve the latitude and longitude of the event address, which are passed into the map.addMarker method to indicate the exact venue location. the map.moveCamera method is also used to faciliate free camera movement and zooming, and the map.setMapType method is used to change the map type visuals.

maps navigation

the latitude and longtitude retrieved from the geocoding api are passed into a google.navigation uri, which opens the google maps app (if installed on the device) and automatically sets the location as the navigation destination.

nearby places

using the google places api, the venue latitude and longtitude (taken from the geocoding api) as well as a chosen proximity radius and selected place type (restaurant, public transit etc) are passed into the placesService getNearbyPlaces method. the latitude and longtitude are then retrieved from each location in the response, and the map.addMarker method is used to set markers on all retrieved locations.

Movie-Actor Database C++ Console App

c++

a c++ console app connected to a folder of csv files holding mock movie and actor data, created for a data structures and algorithms module.

working in a pair, we built the app on c++, using the fstream and sstream libraries to read and write data from the csv files. we created our own data structures, and utilised well-known algorithms such as mergesort to store, retrieve and manipulate the data.

actors and movies are stored in hash tables, with unique ids as keys. actors contain actor data, in addition to a linked list of movie ids (of movies they starred in) and a linked list of ratings. similarly, movies contain movie data, in addition to a linked list of actor ids (of starring actors) and a linked list of ratings.

linked list

a pointer-based linked list, utilising templates to facilitate general use.

comprises the following methods:

  • bool add(const T& item);
  • void remove(int index);
  • const T& get(int index) const;
  • void print() const;
  • bool isEmpty() const;
  • int getLength() const;

hash table

a pointer-based hash table, with linked lists to connect nodes when collisions occur. utilises templates to faciliate general use.

comprises the following methods:

  • bool add(const KeyType& key, ValueType* newItem);
  • bool remove(const KeyType& key);
  • ValueType* get(const KeyType& key) const;
  • bool isEmpty() const;
  • int getLength() const;
  • List> getAllItemsWithKeys() const;
  • void print();
  • bool replace(const KeyType& key, ValueType* newItem);

add actor/movie

the method generateUniqueId, using the c++ random library, is used to generate a random integer, which is checked against the returned bool from the hash table's add method to verify whether it is a unique key.

using cin, the user is then prompted for the actor/movie details. error handling is included, checking for correct data types and logical inputs, with a loop to automatically continue prompting until the user exits with 0.

upon completion, the updateActorsCSV/updateMoviesCSV is called to write the updated data in the respective hash tables to the CSV file.

add actor to movie

the hash table's print function goes through all objects held inside the table, and calls their respective print() function. the iomanip library is used to format and uniformly space out the text display, for ideal viewing experience.

using this function, all movies and actors stored within the database are displayed. from this, the user is prompted to select the actor and movie to add to. the actor and movie are retrieved from the hash table using the get(id) function (time complexity O(1)), and the addMovie/addActor methods are called to add the movie and actor ids to the respective linked lists within the objects.

error handling is included to check for correct data types and correct ID inputs.

upon completion, the updateCastCSV is called to write the updated data in the respective hash tables to the CSV file.

update actor/movie

the hash table's print function is again used to display all movies and actors. similarly, the user is prompted to select the actor/movie using the id.

upon retrieval of the actor/movie using the get(id) function, a while loop will display a menu allowing the user to select which attribute they would like to update. after input, setters in the respective object are used to update the data. the loop will end when the user exits with 0.

error handling is included to check for correct data types and correct ID inputs.

upon completion, the updateActorsCSV/updateMoviesCSV is called to write the updated data in the respective hash tables to the CSV file.
